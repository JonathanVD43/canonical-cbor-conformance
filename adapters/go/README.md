# Go adapter

Implements the `encode` and `decode-strict` modes (each with `--profile
rfc8949` and `--profile dcbor`) of the adapter contract (`CONTRIBUTING.md`),
ported from `adapters/kotlin`/`adapters/typescript` to serve as this
project's fourth reference adapter. Stdlib only -- no third-party module is
imported (`go.mod` declares no `require`s).

## No build framework

`go build` produces a single binary. There's no separate install-dist step
like Kotlin's Gradle build; `harness/adapters.json`'s `go` entry points
straight at the built binary (`adapters/go/adapter`).

## rfc8949 and dcbor profiles: no stdlib CBOR library

Like the Kotlin and TypeScript adapters, both `rfc8949.go` (P1-P9) and
`dcbor.go` (D1-D9) are from-scratch reimplementations against `SPEC.md`, not
library wrappers. `dcbor.go` reuses `rfc8949.go`'s `encodeHead`, `encodeInt`,
`doubleToBeBytes`, `compareBytesUnsigned`, and `bignumTagAndBytes` helpers
directly (same-package top-level functions, since Go has no
`internal`-vs-`private` split within a single package).

Half-precision float conversion (`float16.go`) is also hand-rolled: Go's
stdlib has no float16 type, so `doubleToF16Bits`/`f16BitsToDouble` implement
the same round-to-nearest-even algorithm as the Rust adapter's `half` crate
and the Kotlin/TypeScript adapters' `Float16.kt`/`float16.ts`.

## D8 (NFC normalization): hand-rolled, not a stdlib API

Unlike Java's `java.text.Normalizer`, JS's `String.prototype.normalize`, or
Rust's `unicode-normalization` crate, **Go's standard library has no Unicode
normalization API**, and `golang.org/x/text` is a separate module outside
the stdlib distribution -- so to keep this adapter stdlib-only per
CONTRIBUTING.md, `nfc.go` reimplements Unicode Standard Annex #15's
canonical decomposition + canonical ordering + canonical composition
algorithm directly, against data tables generated by
`scripts/gen_nfc_tables.py` into `nfc_tables.go` (2061 decomposition
entries, 912 combining-class entries, 941 verified composition pairs,
covering Unicode 14.0.0 minus Hangul syllables, which are handled
algorithmically per the standard arithmetic Hangul rule instead of via an
11172-entry table). Each generated composition pair is verified against
Python's own `unicodedata.normalize('NFC', ...)` output (ground truth for
the official composition-exclusion list) rather than the exclusion list
being hand-transcribed. This is the one piece of this adapter that isn't a
direct, mechanical Kotlin/TypeScript port -- see `nfc_test.go` for coverage
(combining-accent composition, Hangul, already-normalized passthrough, and
canonical-ordering-before-composition with reordered combining marks).

## decode-strict mode

`decode.go` is a from-scratch recursive-descent CBOR parser (not built on
any generic decoder, which would normalize away exactly the encoding-choice
metadata -- additional-info width, raw NaN bit pattern, raw map-key byte
spans -- that strict-decode rejection needs to inspect). It implements all
11 reason codes from `SPEC.md`: `NON_SHORTEST_INT`, `UNSORTED_MAP_KEYS`,
`INDEFINITE_LENGTH`, `DUPLICATE_KEY`, `NON_SHORTEST_FLOAT`,
`NAN_PAYLOAD_VARIANT`, `TRAILING_BYTES`, `MULTIPLE_TOP_LEVEL_ITEMS`,
`UNKNOWN_TAG` (both profiles), plus `NON_NFC_STRING` and `UNREDUCED_NUMERIC`
(dcbor-only, since rfc8949 has no NFC or numeric-reduction rule). The parser
returns the first, most specific violation found during descent rather than
enumerating every violation, matching the vector corpus's singular
`expected_reason` contract.

Each stdin line is one hex-encoded raw CBOR byte string; each stdout line is
`ACCEPT <hex>` (the re-encoded canonical form, which must be byte-identical
to the input) or `REJECT <REASON_CODE>`.

`UNKNOWN_TAG`'s allow-list (`allowedTags` in `decode.go`) currently covers
only the bignum tags (2, 3), since that's all the vector corpus exercises
today -- grow it if the corpus adds other tags.

**Known scope gap, replicated deliberately from the Rust/Kotlin/TypeScript
adapters:** a bignum (tag 2/3) whose magnitude fits the native 64-bit range
is not rejected by decode-strict, even though the encoder side requires it
to be a plain int. No documented reason code covers this case and no vector
exercises it.

## Building

```bash
cd adapters/go
go build -o adapter .
```

Produces the binary at `adapters/go/adapter`, matching the harness's
`[binary_path, mode, "--profile", profile]` argv contract
(`harness/adapters.json`'s `go` entry points here).

## Testing

```bash
cd adapters/go
go test ./...
```

## Regenerating the NFC tables

Only needed if targeting a newer Unicode version. Requires Python 3 (stdlib
`unicodedata` only):

```bash
python3 adapters/go/scripts/gen_nfc_tables.py > adapters/go/nfc_tables.go
```
